Initial database schema
First Normal Form (1NF)
Rule of 1NF
No repeating groups
All attributes are atomic (single value per cell)
Each table has a primary key
What changes here?
Your tables are already close to 1NF, but we must:
Ensure each row represents one fact
Add proper primary keys
Remove calculated fields if possible (optional but recommended)
1NF Result
Daily_Record
| SaleID (PK) | Date | Product name | Unit size | Units sold | Price per unit | Total |
(Still has redundancy, but atomic ✔)
Supplier_Product
| Product name | Supplier name |
Supplier
| Supplier name | Phone number | Contact name |
What problems remain?
Product name determines supplier
Supplier name determines phone & contact
Price depends on product + unit size
Massive data repetition
Second Normal Form (2NF)
Rule of 2NF
Must be in 1NF
No partial dependency on a composite key
Key insight here
In Daily_Record, the real natural key is something like:
(Date, Product name, Unit size)
But:
Product name → Supplier name
(Product name, Unit size) → Price per unit
These are partial dependencies.
Fixes in 2NF
We separate data so that:
Product data is not mixed with sales data
Pricing depends only on product + unit size
2NF Resulting Tables
Sale
| SaleID (PK) | Date |
Sale_Item
| SaleItemID (PK) | SaleID (FK) | ProductID (FK) | UnitSizeID (FK) | Units sold |
Product
| ProductID (PK) | Product name | Supplier name |
Product_Unit_Price
| ProductID (FK) | UnitSizeID (FK) | Price per unit |PK: (ProductID, UnitSizeID)
Supplier
| Supplier name (PK) | Phone number | Contact name |
What improved?
Prices are no longer duplicated per sale
Unit-size pricing is handled properly
Still one issue…
Third Normal Form (3NF)
Rule of 3NF
Must be in 2NF
No transitive dependencies(Non-key → Non-key)
Transitive dependencies found
ProductID → Supplier name
Supplier name → Phone number, Contact name
So:
ProductID → SupplierID → Phone number
That violates 3NF.
Fixes in 3NF
Separate Supplier fully
Reference suppliers by ID
Remove derived fields (Total)
Final 3NF Design
Supplier
| SupplierID (PK) | Supplier name | Phone number | Contact name |
Product
| ProductID (PK) | Product name | SupplierID (FK) |
Unit_Size
| UnitSizeID (PK) | Size description |(e.g., “10 pills”, “1 bottle”, “100 ml”)
Product_Unit_Price
| ProductID (FK) | UnitSizeID (FK) | Price per unit |PK: (ProductID, UnitSizeID)
Sale
| SaleID (PK) | Date |
Sale_Item
| SaleItemID (PK) | SaleID (FK) | ProductID (FK) | UnitSizeID (FK) | Units sold |
Total is calculated when needed:
Units sold × Price per unit
Normalization summary
The pharmacy database initially contained sales, product, pricing, and supplier information with significant redundancy and dependency issues. Product and supplier details were repeatedly stored, prices varied by unit size, and derived values such as totals were saved, risking data inconsistency.
First Normal Form (1NF) was achieved by ensuring all attributes were atomic, removing repeating groups, and assigning primary keys. Each record represented a single sale item, but redundancy and dependency problems remained.
Second Normal Form (2NF) addressed partial dependencies by separating sales data from product and pricing data. Prices were stored based on product and unit size, and product details were removed from transaction records. This eliminated partial dependencies but not transitive ones.
Third Normal Form (3NF) removed transitive dependencies by separating supplier information into its own table and linking products to suppliers using foreign keys. Derived attributes, such as total price, were removed and calculated when needed.
The final 3NF design minimizes redundancy, prevents update anomalies, and ensures data integrity while supporting flexible unit-based pricing.

3NF SQL
1 Supplier
CREATE TABLE Supplier (
    supplier_id INT PRIMARY KEY AUTO_INCREMENT,
    supplier_name VARCHAR(100) NOT NULL UNIQUE,
    phone_number VARCHAR(20),
    contact_name VARCHAR(100)
);
2 Product
CREATE TABLE Product (
    product_id INT PRIMARY KEY AUTO_INCREMENT,
    product_name VARCHAR(100) NOT NULL UNIQUE,
    supplier_id INT NOT NULL,
    FOREIGN KEY (supplier_id) REFERENCES Supplier(supplier_id)
);
3 Unit Size
CREATE TABLE Unit_Size (
    unit_size_id INT PRIMARY KEY AUTO_INCREMENT,
    size_description VARCHAR(50) NOT NULL UNIQUE
);
4 Product Unit Price
CREATE TABLE Product_Unit_Price (
    product_id INT NOT NULL,
    unit_size_id INT NOT NULL,
    price_per_unit DECIMAL(10,2) NOT NULL,
    PRIMARY KEY (product_id, unit_size_id),
    FOREIGN KEY (product_id) REFERENCES Product(product_id),
    FOREIGN KEY (unit_size_id) REFERENCES Unit_Size(unit_size_id)
);
5 Sale
CREATE TABLE Sale (
    sale_id INT PRIMARY KEY AUTO_INCREMENT,
    sale_date DATE NOT NULL
);
6 Sale Item (stores price at time of sale)
CREATE TABLE Sale_Item (
    sale_item_id INT PRIMARY KEY AUTO_INCREMENT,
    sale_id INT NOT NULL,
    product_id INT NOT NULL,
    unit_size_id INT NOT NULL,
    units_sold INT NOT NULL CHECK (units_sold > 0),
    unit_price DECIMAL(10,2) NOT NULL,
    FOREIGN KEY (sale_id) REFERENCES Sale(sale_id),
    FOREIGN KEY (product_id) REFERENCES Product(product_id),
    FOREIGN KEY (unit_size_id) REFERENCES Unit_Size(unit_size_id)
);
7 Stock (with expiry tracking)
CREATE TABLE Stock (
    stock_id INT PRIMARY KEY AUTO_INCREMENT,
    product_id INT NOT NULL,
    unit_size_id INT NOT NULL,
    quantity INT NOT NULL CHECK (quantity >= 0),
    expiry_date DATE NOT NULL,
    last_updated TIMESTAMP DEFAULT CURRENT_TIMESTAMP 
                 ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (product_id) REFERENCES Product(product_id),
    FOREIGN KEY (unit_size_id) REFERENCES Unit_Size(unit_size_id)
);
8 Symptom
CREATE TABLE Symptom (
    symptom_id INT PRIMARY KEY AUTO_INCREMENT,
    symptom_name VARCHAR(100) NOT NULL UNIQUE
);
9 Product_Symptom (Many-to-Many)
CREATE TABLE Product_Symptom (
    product_id INT NOT NULL,
    symptom_id INT NOT NULL,
    PRIMARY KEY (product_id, symptom_id),
    FOREIGN KEY (product_id) REFERENCES Product(product_id)
        ON DELETE CASCADE,
    FOREIGN KEY (symptom_id) REFERENCES Symptom(symptom_id)
        ON DELETE CASCADE
);
10 Role
CREATE TABLE Role (
    role_id INT PRIMARY KEY AUTO_INCREMENT,
    role_name VARCHAR(50) NOT NULL UNIQUE
);
11 Users (renamed from User because USER is reserved)
CREATE TABLE Users (
    user_id INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    full_name VARCHAR(100) NOT NULL,
    role_id INT NOT NULL,
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    FOREIGN KEY (role_id) REFERENCES Role(role_id)
);

Relationship explaination
1️ Role → User (One-to-Many)
One role can have many users.
Example:
Role: "Pharmacist"
Users: John, Maria, Alex
This supports role-based access control.
2️ Supplier → Product (One-to-Many)
One supplier supplies many products.
One product belongs to one supplier.
Example:
Supplier A supplies:
Paracetamol
Amoxicillin
Cough Syrup
3️ Product → Unit_Size (Many-to-Many via Product_Unit_Price)
A product can have multiple sizes.
Example:Paracetamol:
500mg tablet
250mg tablet
100ml syrup
Each size has its own price.
That is why we use:
Product_Unit_Price(product_id, unit_size_id)
This ensures:
No duplicate pricing
Proper normalization
Flexible variant management
4️ Product → Stock (One-to-Many)
One product can have multiple stock batches.
Example:Paracetamol 500mg:
Batch A – Exp: 2026-01-01 – Qty: 100
Batch B – Exp: 2026-03-01 – Qty: 200
This is critical for pharmacy because:
Each batch expires differently
Must use FEFO (First Expire First Out)
5️ Sale → Sale_Item (One-to-Many)
One sale can contain multiple products.
Example:
Sale #101:
2 Paracetamol
1 Cough Syrup
Each is stored in Sale_Item.
6️ Sale_Item → Stock (Many-to-One)
Each sale item deducts from a specific stock batch.
This ensures:
Proper expiry tracking
Correct inventory deduction
Batch accountability
7️ Product ↔ Symptom (Many-to-Many)
A product can treat multiple symptoms.A symptom can have multiple drug options.
Example:Symptom: FeverDrugs:
Paracetamol
Ibuprofen
This supports:
✔ Search drug by symptom✔ Suggest alternatives

1NF Rules:
No repeating groups
Atomic values (no multiple values per cell)
Primary key exists
You turned the notebook into something like:

Problems Still Present

(Product, UnitSize) → Price
Total is calculated (Units × Price)
Still heavy redundancy.
Step 2 — Second Normal Form (2NF)
2NF Rule:
Must be in 1NF
No partial dependency on composite key
The real natural key of sales was:
(Date, Product, UnitSize)
But:
Product → Supplier
(Product, UnitSize) → Price
These are partial dependencies, because:They depend only on part of the key.
Fix in 2NF
We separate:

Supplier
supplier_name | phone | contact
Improvements
Price no longer repeated in sales
Sales no longer store supplier details
Less duplication
But…
Step 3 — Third Normal Form (3NF)
3NF Rule:
Must be in 2NF
No transitive dependency
Non-key attribute must depend only on the key
Transitive Dependency Problem
In 2NF design:
ProductID → SupplierName
SupplierName → Phone, Contact
Therefore:
ProductID → Phone
This is a transitive dependency.
That violates 3NF.

Separate Supplier fully and use IDs.

phone_number
contact_name

Now:
Product depends only on its key
Supplier info depends only on supplier_id
No transitive dependency
✔ 3NF satisfied

3NF check proof
A relation is in Third Normal Form (3NF) if:
It is in 1NF
It is in 2NF
For every functional dependency:
X → Y
At least one of the following is true:
X is a superkey, OR
Y is a prime attribute (part of some candidate key)
In simpler words:
Every non-key attribute must depend on the key, the whole key, and nothing but the key.
